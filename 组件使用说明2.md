# 工业缺陷检测管理系统组件使用说明（二）

## 1. 文档定位

本文档用于在项目展示、课程设计与竞赛答辩中，快速说明系统组件构成、技术路线、调用关系与合规性信息。  
相较于基础版说明，本文件补充了应用场景、业务价值与验收关注点，便于评审理解系统完整性。

## 2. 系统目标与应用场景

### 2.1 系统目标

- 建立缺陷从创建到闭环的全流程管理机制。
- 降低纸面登记和人工追踪成本，提升处理效率与可追溯性。
- 支持多角色协作（管理员、检验员、查看者）并提供审计日志。

### 2.2 应用场景

- 生产现场缺陷发现与上报
- 质量管理部门指派与处理跟踪
- 验收与关闭环节的责任留痕
- 过程复盘与质量趋势统计分析

## 3. 技术栈与架构声明

- 前端：`React`（Create React App）+ `Ant Design` + `react-router-dom`
- 后端：`Go`（Golang）标准库 `net/http`
- 数据库：`SQLite`
- 通信方式：前端直接调用后端 REST API（`/api/*`）
- 架构说明：后端不依赖 Gin、Echo、Beego 等第三方 Web 框架，核心服务由 Go 标准库与业务代码实现

## 4. 开源合规性说明

- 项目使用的依赖来自公开开源生态（npm、Go Modules）。
- 依赖使用遵循相应许可证要求，满足开源合规管理要求。
- 建议交付时提供依赖清单与许可证归档，便于审计与复核。

## 5. 前端组件使用说明（React）

### 5.1 通用组件

#### `DefectList.js`

- 作用：展示缺陷数据列表，承载筛选与操作入口。
- 输入：列表数据、查询条件、回调函数。
- 输出：用户操作事件（查看、编辑、流程动作触发等）。

#### `DefectForm.js`

- 作用：缺陷创建与编辑表单。
- 输入：初始数据、可选项数据（如缺陷类型、设备信息）。
- 输出：结构化表单数据，提交给后端接口。

#### `ProtectedRoute.js`

- 作用：路由级鉴权与权限控制。
- 输入：当前用户登录状态、角色信息、目标路由。
- 输出：允许访问或重定向至登录/无权限页面。

#### `MasterDataManagePage.js`

- 作用：主数据通用 CRUD 页面骨架。
- 复用对象：产线管理、设备管理、缺陷类型管理。
- 优点：减少重复页面开发，统一交互风格与校验逻辑。

### 5.2 页面组件

- `DashboardPage.js`：统计看板与趋势展示
- `LoginPage.js`：用户登录与会话建立
- `DefectListPage.js`：缺陷查询与流程操作入口
- `DefectDetailPage.js`：缺陷详情、流转记录与处理信息
- `DefectCreatePage.js`：新建缺陷登记
- `ProductionLineManagePage.js`：产线主数据维护
- `EquipmentManagePage.js`：设备主数据维护
- `DefectTypeManagePage.js`：缺陷类型主数据维护
- `UserManagementPage.js`：用户与角色管理
- `OperationLogsPage.js`：操作审计与追溯查看

### 5.3 前端基础模块

- `App.js`：路由配置与页面装配入口
- `index.js`：前端应用启动入口
- `api.js`：统一请求封装（鉴权头、异常处理、接口基址）

## 6. 后端模块使用说明（Go）

### 6.1 `backend/main.go`（核心业务）

- 负责用户认证、缺陷流转、日志、导出等核心接口。
- 基于标准库 `net/http` 实现请求处理与中间逻辑。
- 通过角色权限控制保障操作边界。

### 6.2 `backend/masterdata.go`（主数据业务）

- 负责产线、设备、缺陷类型等主数据接口。
- 提供统一的新增、查询、修改、删除能力。

### 6.3 `backend/main_test.go`（测试）

- 使用 `httptest` 构建接口级测试。
- 验证关键流程的输入输出与状态转换正确性。

## 7. 数据库说明（SQLite）

- 数据库引擎：`SQLite`
- 使用方式：本地嵌入式存储，部署与迁移成本低
- 存储对象：缺陷单、用户、会话、主数据、操作日志、流程日志等
- 适用性：适合课程设计、原型验证与中小规模系统部署

## 8. 组件调用关系

1. 页面组件调用通用组件（表单、列表、权限路由）构建业务页面。
2. 页面通过 `api.js` 发起请求至后端 `/api/*`。
3. 后端模块处理业务逻辑并读写 SQLite。
4. 处理结果以 JSON 返回前端并驱动界面更新。

## 9. 业务价值与验收关注点

### 9.1 业务价值

- 流程标准化：减少口头/纸质流转导致的信息缺失。
- 责任可追溯：全流程日志化，便于复盘审计。
- 决策可视化：通过看板了解缺陷分布与处理趋势。

### 9.2 验收关注点

- 是否完成缺陷闭环（创建、指派、处理、验收、关闭）
- 是否满足角色权限边界与访问控制
- 是否具备日志追溯能力
- 是否保证前后端联调稳定与数据一致性

## 10. 维护建议

- 新增同类型主数据页面时优先复用 `MasterDataManagePage.js`。
- 新增 API 保持 `/api/` 命名统一，便于鉴权与审计扩展。
- 引入新依赖时同步检查许可证并更新合规说明。
- 对外材料统一说明“无第三方 Web 框架依赖”，确保技术口径一致。
